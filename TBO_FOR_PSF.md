# TBO For PSF

## What TBO Is Here

In the PSF repo, TBO means a local operating framework for keeping the project organised.

It is not the Tinyme project.

It is not a hosted service.

It is not a public dependency.

For PSF, the useful part of TBO is the planning behavior: a shared queue, clear ownership, explicit dependencies, and a clean handoff rhythm.

## Why We Use It

PSF will have several active work streams:

- strategy
- content
- marketing
- tech stack
- launch planning
- sponsorships
- domains and IP setup

TBO gives us a way to keep those lanes structured while still working in separate chats.

## What TBO Should Help With

- master planning
- decision capture
- handoffs between chats
- clear update paths
- keeping PSF docs consistent
- telling us what should happen next
- helping the two-person team stay aligned without carrying the whole project in memory

## What TBO Should Not Become

- a second project identity
- a source of Tinyme confusion
- a heavy process layer that slows PSF down
- a reason to over-document simple decisions
- a substitute for the PSF docs themselves

## How To Use It In PSF

- keep `MASTER_PLAN.md` as the main reference
- use `PROJECT_BOUNDARIES.md` to keep PSF separate from Tinyme
- use `PBO_PLAN.md` for the working queue and orchestration rhythm
- update the relevant section in the master plan when a chat reaches a decision
- use short supporting docs only when they help one specific lane

## Update Rule

If a TBO concept helps PSF move faster or stay organised, keep it.

If it makes the repo feel like a different project, leave it out.
